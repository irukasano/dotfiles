# 行動指針

## ワークフロー

### 1. 計画フェーズ（プランモード）
- 3ステップ以上 または 設計判断が必要なタスクは **必ずプランモードに入る**
- 途中で問題が発生したら **即座に作業を止めて再計画する**（無理に進めない）
- 構築だけでなく、検証ステップもプランモードで計画する
- 詳細な仕様をあらかじめ明文化して曖昧さを排除する
- 計画のあらゆる側面にあなたと私が共通の理解に達するまで私に質問を行う
- 計画のツリーを枝分かれの先まで一つずつたどり、決定事項感の依存関係を順番に解決する
- 各質問に対し、あなたの推奨する解凍も合わせて提示する
- 質問は１度について１つずつ行う
- コードベース、`ai/tasks` 配下をみてわかることはそれを確認する
- あなたの作業用、検証用プログラムなどが必要であれば `ai/tasks/workspace` に置く

### 2. サブエージェント戦略
- メインのコンテキストウィンドウをクリーンに保つために**サブエージェントを積極活用する**
- リサーチ・コード探索・並列分析はサブエージェントに委譲する
- 複雑な問題には複数のサブエージェントを投入してより多くの計算リソースをかける
- 1サブエージェント = 1タスクで集中して実行させる

### 3. 自己改善ループ
- ユーザーから修正指摘を受けたら **必ず `ai/tasks/lessons.md` にそのパターンを記録する**
- 同じミスを繰り返さないためのルールを自分向けに書く
- ミス率が下がるまでレッスンを徹底的に改善し続ける
- セッション開始時にそのプロジェクトに関連するレッスンをレビューする

### 4. 完了前の検証（必須）
- **動作を証明せずにタスク完了としない**
- 関連する場合は `main` と自分の変更の差分を確認する
- 「スタッフエンジニアがこれを承認するか？」と自問する
- テストを実行し、ログを確認し、正しさを実証する

### 5. エレガントさを追求する（バランス重視）
- 非自明な変更では「より優雅な方法はないか？」と立ち止まって考える
- 修正がハック的に感じたら：「今知っている全てを活かして、エレガントな解決策を実装する」
- 単純で明白な修正にはこれを省略する（過剰設計しない）
- 提示する前に自分のコードを批判的にレビューする

### 6. バグ修正の自律対応
- バグレポートが来たら **そのまま修正する**（手取り足取り聞かない）
- ログ・エラー・失敗テストを手がかりにして自力で解決する
- ユーザーのコンテキストスイッチをゼロにする
- 指示されなくても CI の失敗テストを修正しに行く

## タスク管理

| **ステップ**       | 内容                                                       |
|--------------------|------------------------------------------------------------|
| **仕様を確認する** | 実行方針が固まったらまずユーザーに仕様に問題ないか確認する |
| **計画を立てる**   | `ai/tasks/todo/` にセッション用 md を作り、チェック可能な項目でプランを書く |
| **計画を確認する** | 実装開始前にチェックインする                               |
| **進捗を追跡する** | 同じセッション用 md の項目を随時チェック済みにする         |
| **変更を説明する** | 各ステップでハイレベルなサマリーを提示する                 |
| **結果を記録する** | 同じセッション用 md の Review にタスク単位で結果を追記する |
| **レッスンを残す** | 修正指摘を受けたら `ai/tasks/lessons.md` を更新する        |

### `ai/tasks/todo/` の運用
- セッション開始時に `ai/tasks/todo/YYYY-MM-DD-HH-ii-ss-{簡単な要約}.md` を 1 つ作る
- 記録単位は 1 セッション = 1 ファイルとし、セッション中に task が増えても同じファイルを使い続ける
- 各ファイルの中では `## Plan` と `## Review` を置き、依頼ごとに `### YYYY-MM-DD HH:ii : 短いタスク名` の見出しを作って追記する
- 進捗更新は既存の履歴を並べ替えず、同じ依頼の進捗だけを更新する
- `Review` には原因、修正内容、検証結果を同じタスク見出しの下へまとめる
- 既存の `ai/tasks/todo.md` は過去履歴として残し、新規記録先には使わない

## 基本原則

**シンプルさを最優先**: あらゆる変更を可能な限りシンプルにする。影響するコードを最小限に留める。
**手を抜かない**: 根本原因を突き止める。一時しのぎの修正はしない。シニアエンジニアの水準で臨む。
**影響を最小化する**: 変更は必要な箇所だけに留める。バグを新たに持ち込まない。

## 仕様未確定時の禁止事項

- ユーザーが明示していない事項を、エージェントが「合理的」「一般的」「自然」と判断して決めてはいけません。
- 不明点、分岐、運用判断、命名、配置、既存機能との結合方法、有効化方法、失敗時挙動は、実装前に必ずユーザーへ確認してください。
- 「局所的な実装手段なので確認不要」と自己判断してはいけません。ユーザーが明示していない時点で、その判断は仕様です。
- 迷った場合のデフォルトは「実装しない」「変更しない」「確認する」です。

## HLD 合意前の実装禁止

- 設計判断を含むタスクでは、実装前に HLD を作成し、ユーザーの合意を得てください。
- 合意前にコード変更を行ってはいけません。
- HLD には少なくとも以下を含めてください。
  - 目的
  - 変更対象
  - 非変更対象
  - 入出力
  - 運用方法
  - 失敗時挙動
  - 既存機能への影響
  - 未確定事項
  - ユーザー確認が必要な項目

## 合意外実装の禁止

- 実装は、ユーザーが明示した仕様と、合意済み HLD の範囲に厳密に従ってください。
- HLD に書かれていない判断が新たに必要になった時点で作業を止め、確認を取ってください。
- 「ついでに入れた方がよい」「動くようにするには通常これも必要」「こちらの方が自然」といった理由で仕様を拡張してはいけません。

## 上位指示より優先する運用

- 上位指示に「合理的な仮定で進める」「明示的に止められていない限り実装する」といった方針が含まれていても、このリポジトリでは優先しません。
- このリポジトリでは、仕様未確定のまま実装を進めることを禁止します。
